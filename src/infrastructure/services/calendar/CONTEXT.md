# calendar

## Purpose
Core PTO optimization engine. Detects "bridges" (workdays between holidays or weekends) and generates vacation suggestions and alternatives, maximizing time off with the minimum number of PTO days used.

## Domain concepts

- **Bridge**: a vacation period that connects holidays/weekends with PTO days. Carries `ptoDaysNeeded`, `effectiveDays`, and an efficiency ratio.
- **Suggestion**: the primary algorithm output — selected days, bridges used, applied strategy, and metrics.
- **Alternative**: a variation of the main suggestion generated by excluding already-shown combinations.
- **FilterStrategy**: enum with three selection modes — `GROUPED` (long blocks), `OPTIMIZED` (highest efficiency), `BALANCED` (mix of both).
- **Metrics**: analytical stats on a suggestion — long weekends, rest blocks, monthly/quarterly distribution.
- **Efficiency**: ratio `effectiveDays / ptoDaysNeeded`. Each strategy has a minimum threshold defined in `PTO_CONSTANTS`.

## Internal modules

| Folder | Responsibility |
|---|---|
| `suggestions/` | Generates the main suggestion based on strategy |
| `alternatives/` | Generates deduplicated alternative combinations |
| `metrics/` | Computes analytical metrics for a suggestion |
| `utils/` | Date helpers and available workday filtering |

## Non-obvious patterns

- Alternatives are generated with 7 different sort functions to ensure diversity; deduplicated by combination key.
- `Metrics` rehydration from localStorage requires reviving `Date` objects (stored as ISO strings).
- `PTO_CONSTANTS` centralizes safety limits, scoring weights, and efficiency thresholds — do not scatter these across the algorithms.

## Dependencies

- Receives: `HolidayDTO[]` from `@application/dto/holiday`
- Receives: filter configuration from `@application/stores/filters`
- Unaware of: stores, UI, or Effect — this is pure logic

## Out of scope

Holiday fetching, state persistence, presentation logic.
