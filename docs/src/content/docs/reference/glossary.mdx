The suggestion-engine terms below are defined in code in `src/domain/calendar/types.ts` and explained in depth in `src/domain/calendar/CONTEXT.md`.

### PTO

Paid time off — the user's budget of vacation days for the planning period (`ptoDays` in the filters store, `src/application/stores/filters.ts`). Everything the engine does is about spending this budget as efficiently as possible.

### Bridge

A block of PTO days that connects weekends and public holidays into one contiguous stretch off. A single PTO day on a Monday before a Tuesday holiday "bridges" the weekend to the holiday, yielding four days off for one day spent. Modeled by the `Bridge` interface: `ptoDaysNeeded`, `effectiveDays`, `efficiency` and the concrete `ptoDays` dates. A **bridge day** is one of the PTO days inside a bridge.

### Effective days

The total consecutive days off a bridge or suggestion produces — PTO days plus the weekends and holidays they absorb. **Efficiency** is `effectiveDays / ptoDaysNeeded`; the engine discards bridges below a minimum efficiency threshold (tunables in `src/domain/calendar/const.ts`).

### FilterStrategy

How bridges are ranked and selected. Three values, exported as the `FilterStrategy` const object in `src/domain/calendar/types.ts`:

- **`grouped`** — prefer multi-day blocks, with efficiency as the tiebreaker. Fewer, longer breaks.
- **`optimized`** — prefer the highest-efficiency bridges first. Maximum days off per PTO day.
- **`balanced`** — two-pass greedy scoring that prioritizes high-value long bridges, then fills the remaining budget.

### Suggestion

The engine's output for a given budget, calendar and strategy: the set of recommended PTO dates plus the bridges and metrics behind them (`Suggestion` interface, produced by `src/domain/calendar/suggestions/generateSuggestions.ts`). Users can hand-edit the suggested days afterwards.

### Alternatives

Distinct alternative PTO plans for the same inputs, produced by `src/domain/calendar/alternatives/generateAlternatives.ts` using different bridge-ordering strategies — so the user can compare plans rather than accept a single answer.

### Carry-over

Months of the following year included in the planning window (`carryOverMonths` in the filters store). It lets a plan spill unused PTO past the year boundary — e.g. bridging the winter holidays into January.

### Premium key

The identifier of a premium purchase — the Stripe PaymentIntent id, surfaced as `premiumKey` (see `src/application/use-cases/activatePremium.ts` and the premium store `src/application/stores/premium.ts`). The active session itself is carried by the `premium-token` JWT cookie — see [cookies](/reference/cookies/).
