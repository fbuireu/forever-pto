`src/middleware.ts` is the single middleware entry point. It runs a short, strictly ordered chain and returns as early as it can. Each step delegates to a focused helper — the middleware file itself is orchestration only.

## The matcher

```ts
export const config = {
  matcher: ['/((?!api|_next|_vercel|.*\\..*).*)', '/api/markdown'],
};
```

Everything under `/api`, Next internals (`/_next`, `/_vercel`) and any path with a file extension are skipped — with one deliberate exception: `/api/markdown` is matched so it can receive a cache header (step 1).

## The chain, in order

1. **`/api/markdown` cache header.** Requests to the markdown route itself pass through untouched except for a `Cache-Control` header (`MARKDOWN_CACHE_CONTROL` in `src/middleware.ts`, one hour public caching).

2. **Markdown content negotiation.** If the request carries `Accept: text/markdown` and the path is not internal (`/api/*`, `/.well-known/*`), the request is rewritten to `/api/markdown?path=<pathname>` with the same cache header. The route handler (`src/app/api/markdown/route.ts`) builds a markdown rendition of the page via `src/infrastructure/markdown/` — this is how agents and LLMs read the site as text.

3. **Payment-confirmation guard.** Any path ending in `/payment/confirmation` without a `payment_intent` search param is redirected to the locale home. The confirmation page can therefore assume the param exists.

4. **i18n.** `createMiddleware(routing)` from `next-intl/middleware` resolves the locale (routing config in `src/infrastructure/i18n/routing.ts`). If next-intl set the locale cookie (`LOCALE_COOKIE`, exported from `src/infrastructure/i18n/config.ts`), the middleware re-sets it via `setLocaleCookie` from `src/infrastructure/i18n/cookie.ts` to harden its attributes. Cookie details are catalogued in the [cookies reference](/reference/cookies/).

5. **Country detection — always last.** The i18n response is passed through `location()` from `src/infrastructure/proxy/location.ts`, which runs `detectCountry(request)` and, when a country is found, sets the `USER_COUNTRY_COOKIE` cookie via `src/infrastructure/proxy/cookie.ts`. This cookie is intentionally readable from client JS — the location store uses it to preselect the user's country. The detection strategy chain (Cloudflare trace → `cf-ipcountry` header → IP lookup) is documented in [country detection](/how-it-works/country-detection/).

Running country detection last matters: it decorates whichever response the earlier steps produced (rewrite, redirect or the i18n response) instead of forcing its own.

## Why proxy helpers live in infrastructure

The middleware imports nothing framework-agnostic itself — the actual logic (`detectCountry`, cookie writers) lives under `src/infrastructure/proxy/` and `src/infrastructure/services/location/`, keeping `src/middleware.ts` a readable table of contents for the request pipeline and keeping the helpers unit-testable (`src/middleware.test.ts` covers the orchestration).
