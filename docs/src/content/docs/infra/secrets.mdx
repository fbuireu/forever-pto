There are three distinct classes of configuration, and each has exactly one home. Mixing them up is the main way deploys break.

| Class | Lives in | Reaches the app as |
| --- | --- | --- |
| Build-time public vars (`NEXT_PUBLIC_*` tokens/ids) | GitHub environment **variables** | Baked into the client bundle by `cf:build` |
| Runtime secrets | GitHub environment **secrets** → `wrangler secret bulk` | Worker-attached secrets (`process.env` at runtime) |
| Non-secret runtime vars | `[vars]` / `[env.*.vars]` in `wrangler.toml` | Worker vars, committed and versioned |

## GitHub environments

Two GitHub environments, `development` and `production`, hold both classes of per-environment values:

- **Variables** (`vars.*`): the `NEXT_PUBLIC_*` values (Better Stack tokens, Stripe publishable key, GA id, storage key) that `_deploy.yml` injects into the build step. Being `NEXT_PUBLIC_*`, they end up in the public client bundle — they are *configuration*, not secrets, but still per-environment.
- **Secrets**: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, and the five runtime secrets below, plus `CF_ACCESS_CLIENT_ID` / `CF_ACCESS_CLIENT_SECRET` for the E2E job. Repo-level secrets also include `PAT` (releases and bot approvals) and `CODECOV_TOKEN`.

This documentation site needs none of the app's values — its CI deliberately uses no GitHub environment secrets beyond the Cloudflare deploy credentials (see [docs CI](/infra/docs-ci/)).

## Runtime secrets: pushed on every deploy

The five runtime secrets — `JWT_SECRET`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `RESEND_API_KEY`, `TURSO_AUTH_TOKEN` — are **re-pushed on every deploy** by the last step of `.github/workflows/_deploy.yml`:

1. A Node one-liner validates that all five are present, failing the job before anything half-configured ships.
2. The values are piped as JSON into `pnpm exec wrangler secret bulk`, targeting `--name pr-<n>-…` for per-PR previews (a freshly created Worker has no secrets at all) or `--env <environment>` for the named Workers.

Re-pushing every time makes deploys idempotent: rotating a secret is "update the GitHub environment secret, redeploy". Preview Workers take their secrets to the grave when [cleanup deletes them](/infra/environments/).

## Local development

- `.dev.vars` (gitignored) holds the local equivalents of the Worker secrets; wrangler and `initOpenNextCloudflareForDev()` pick it up automatically. `.env.example` documents the expected keys.
- **Never commit `.env` files.** The repo currently has committed `.env.development` / `.env.production` files containing real-looking credentials — a known hygiene issue with a pending follow-up (rotate + purge). Do not add to them, do not read them from new tooling, and never let the docs build touch them.

## Conventions

- New runtime secret? Add it in four places: the GitHub environments, the `secrets:` contract of `_deploy.yml` (inputs *and* the bulk-push list), `.env.example`, and `environment.d.ts` for typing.
- New public build var? GitHub environment **variable** + the `env:` block of the build step in `_deploy.yml`.
- Non-secret, per-environment runtime value (URLs, feature toggles)? `wrangler.toml` vars — committed, reviewed, no ceremony.
